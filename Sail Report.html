Purpose

What SAIL is intended to solve

Several operational workflows are currently split across tools, memory, and manual handoffs. Printed parts, QC decisions, scans, refurbishment actions, assembly consumption, and fleet context all affect each other, but they are rarely in one place that technicians can actually trust.

SAIL is meant to be that connective layer. It gives parts a visible lifecycle, gives technicians one place to work, and creates a cleaner line between what is happening on the floor and what the software reflects.

Operational goal: reduce drift between what is happening on the floor and what the software says is true.
User goal: make the system fast to understand and practical enough that technicians will keep using it.
Management goal: improve traceability, workflow clarity, and confidence in inventory and maintenance state.

Architecture

How the system is structured

LAYER 01

Upstream operational source

  • Read-only manufacturing and fleet data
  • Master references, BOMs, robots, and supporting context
  • Never mutated directly by SAIL
LAYER 02

SAIL execution layer

  • Writable system for workflow state, overrides, sessions, and user preferences
  • Handles application logic, event logging, and technician actions
  • Controlled boundary, never writes back into upstream data
LAYER 03

Hardware and operator interface

  • React interface for technicians and admins
  • Node bridge for printers, scanners, and hardware diagnostics
  • Physical work and system state stay aligned

Hardest engineering problem

What has been hardest so far

The hardest part has been BOM data integrity in the Assembly Builder.

Some nested BOM data from the upstream source was producing incorrect quantities in certain cases, particularly around fasteners. The interface could be reflecting the source correctly and still be operationally wrong for the technician using it.

I solved it by building a correction layer inside SAIL's own database. That lets me fix source-derived values without touching the upstream system, which keeps the architecture clean while solving the actual floor-level problem.

Technical value already created

What is strongest in the current build

The most valuable part of SAIL right now is the inventory state model running across actual workflows.

Parts move through real operational states and those transitions are reflected in scan flows, QC, refurbishment, assembly, and admin visibility. The system is not just displaying information. It is enforcing process logic and keeping a usable record of what happened.

That foundation is what makes the remaining work worth finishing. There is already a lot of substance underneath.

My recommendation is a Mac mini M2 as the bridge node. It gives me reliable Bluetooth for the Brady label printer, built-in macOS CUPS for print management, solid SSD storage, and a straightforward environment to manage services while I am still actively iterating. A Raspberry Pi is a fine prototyping device but it introduces more setup tradeoffs around storage, Bluetooth stability, and day-to-day convenience than I think are worth it at this stage.

Summary

Why the development push has been heavy

The recent acceleration was intentional. I wanted to get SAIL past the point of being a concept and into a state where the system is real and actually works. That meant investing heavily in architecture, workflow shape, and technical setup up front rather than building thin and patching later.

The benefit is that the remaining work is narrower. The biggest unknowns are no longer about what the product should be. What is left is mostly making the hardware communication reliable, closing bugs, and tightening the final operational experience before the first real review.

My view is that SAIL has already crossed the line from concept to working internal tool. The next step is finishing the communication layer and getting the first controlled version into a state worth evaluating seriously.
